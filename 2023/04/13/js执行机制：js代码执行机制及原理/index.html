# js 的同步与异步

  • JS 的同步与异步是指 JS 的执行模式，也就是 JS 如何处理多个任务的问题。
  • 同步模式就是上一个任务执行完毕，才能执行下一个任务，程序的执行顺序与任务的排列顺序是一致的、同步的；
  • 异步模式则是每一个任务有一个或多个回调函数（callback），上一个任务结束后，不是执行下一个任务，而是执行回调函数，下一个任务则是不等上一个任务结束就执行，所以程序的执行顺序与任务的排列顺序是不一致的、异步的。
  • 异步编程技术使你的程序可以在执行一个可能长期运行的任务的同时继续对其他事件做出反应而不必等待任务完成。

# JS 中主线程的执行栈（栈与执行栈）

  • 栈是一种数据结构，具有先进后出的原则。也就是说，栈中最后进入的元素会最先出来。

  • 主进程的执行栈是指 JavaScript 引擎执行代码时的一个内存空间，用来存放执行上下文。

  • 当引擎第一次遇到 JavaScript 代码时，会产生一个全局执行上下文并压入执行栈，每遇到一个函数调用，就会往栈中压入一个新的上下文。

  • 当函数执行完毕后，就会从栈中弹出对应的上下文，并返回到上一层的上下文继续执行。因此，主进程的执行栈并不是先进先执行，而是按照代码的顺序和函数调用的顺序来执行的。

      • 在 foo 函数中，打印出 "foo"，然后函数结束，从栈中弹出 foo 的执行上下文，并返回到 bar 的执行上下文继续执行。
      • bar 函数也结束了，从栈中弹出 bar 的执行上下文，并返回到 baz 的执行上下文继续执行。
      • baz 函数也结束了，从栈中弹出 baz 的执行上下文，并返回到全局的执行上下文继续执行。
      • 全局代码也结束了，从栈中弹出全局的执行上下文，并结束程序。
      • 可以看出，栈和执行栈都是遵循先进后出的原则的。

# js 异步任务队列

  • JS 异步任务队列是指存放异步任务的一个队列，它的作用是在主线程执行完同步任务后，按照一定的规则执行队列中的异步任务。

  • JS 中的异步任务有多种类型，常见的有以下几种：

    • 宏任务（macro task）：就是那些可以直接进入任务队列的异步任务，比如 setTimeout, setInterval, setImmediate, requestAnimationFrame, I/O 等。
    • 微任务（micro task）：就是那些不能直接进入任务队列，而需要等待当前执行栈清空后才能执行的异步任务，比如 Promise.then, process.nextTick, MutationObserver 等。
    • UI 渲染任务：就是那些涉及到浏览器界面更新的异步任务，比如 DOM 操作，CSS 动画等。

    # JS 异步任务队列的执行规则：

    1. 首先执行主线程上的同步任务。
    2. 然后检查微任务队列，如果有微任务，就依次执行所有微任务，直到清空微任务队列。
    3. 然后取出宏任务队列中的第一个宏任务，执行宏任务。
    4. 然后再次检查微任务队列，重复第二步。
    5. 然后检查是否需要进行 UI 渲染，如果需要，就执行 UI 渲染。
    6. 然后再次取出宏任务队列中的下一个宏任务，重复第三步至第五步，直到宏任务队列清空。

# JS 中常见异步编程的方法

  • JS 中有多种异步编程的方法，常见的有以下几种：

    • 回调函数：就是把一个函数作为参数传递给另一个函数，当异步操作完成后，调用这个函数来处理结果。
    • 事件监听：就是给异步操作绑定一个事件监听器，当异步操作触发了某个事件，就执行相应的事件处理函数。
    • 发布 / 订阅：就是通过一个中间对象来管理事件的订阅和发布，当异步操作完成后，发布一个消息，订阅了这个消息的函数就会被执行。
    • Promise 对象：就是一个代表异步操作的对象，它有三种状态（pending, fulfilled, rejected），可以通过 then 方法来指定状态改变时的回调函数。
    • async/await：就是一种语法糖，可以让异步操作看起来像同步操作一样，它实际上是基于 Promise 实现的。

# Promise.then 和 setTimeout 哪个先执行？

  • Promise.then 和 setTimeout 哪个先执行，取决于它们是在什么时候注册的异步任务。

  • 如果 Promise.then 和 setTimeout 是在同一个宏任务中注册的，那么 Promise.then 会先执行，因为 Promise.then 会进入微任务队列，而 setTimeout 会进入宏任务队列，而微任务队列的优先级高于宏任务队列。

# js 中宏任务与微任务

  • JS 中的宏任务和微任务是指 JS 的异步任务的两种类型，它们与事件循环有关，决定了 JS 的执行顺序。

  • 宏任务（macro task）就是那些可以直接进入任务队列的异步任务，比如 setTimeout, setInterval, setImmediate, requestAnimationFrame, I/O 等。

  • 微任务（micro task）就是那些不能直接进入任务队列，而需要等待当前执行栈清空后才能执行的异步任务，比如 Promise.then, process.nextTick, MutationObserver 等。

    # 宏任务和微任务的执行规则：

    1. 首先执行主线程上的同步任务。
    2. 然后检查微任务队列，如果有微任务，就依次执行所有微任务，直到清空微任务队列。
    3. 然后取出宏任务队列中的第一个宏任务，执行宏任务。
    4. 然后再次检查微任务队列，重复第二步。
    5. 然后检查是否需要进行 UI 渲染，如果需要，就执行 UI 渲染。
    6. 然后再次取出宏任务队列中的下一个宏任务，重复第三步至第五步，直到宏任务队列清空。

  • 区分宏任务和微任务的目的是为了让一些需要在当前执行栈清空后立即执行的异步任务能够及时得到处理，而不用等待下一个宏任务的执行。

  • 这样可以提高代码的执行效率，避免一些不必要的延迟，也可以保证一些状态的同步，比如 DOM 操作，Promise 状态变化等。

# 为什么微任务队列的优先级高于宏任务队列？

  • 微任务队列的优先级高于宏任务队列，是为了让一些需要在当前执行栈清空后立即执行的异步任务能够及时得到处理，而不用等待下一个宏任务的执行。
  • 这样可以提高代码的执行效率，避免一些不必要的延迟，也可以保证一些状态的同步，比如 DOM 操作，Promise 状态变化等。
  • 另外，微任务队列的优先级高于宏任务队列，也是为了与 ES6 的 Promise 规范保持一致，因为 Promise 规定，Promise 的回调函数应该在本轮事件循环的末尾执行，而不是下一轮事件循环的开始。

# 什么是事件循环？(EventLoop)

  • 事件循环是 JS 处理非阻塞 I/O 操作的机制，它可以让 JS 在单线程的情况下，异步地执行多个任务。

  • 事件循环的基本原理是，JS 有一个主线程和一个任务队列，主线程负责执行同步任务，任务队列负责存放异步任务的回调函数。

  • 当主线程执行完同步任务后，就会去读取任务队列中的回调函数，依次执行它们，直到任务队列为空。然后再去执行下一个同步任务，如此循环。

  • 每一次循环就是一个事件周期或称为一次 tick。

  • 任务队列又分为宏任务队列和微任务队列，宏任务队列包括 setTimeout, setInterval, setImmediate, requestAnimationFrame, I/O 等，微任务队列包括 Promise.then, process.nextTick, MutationObserver 等。

  • 事件循环的执行规则是：

    1. 首先执行主线程上的同步任务。
    2. 然后检查微任务队列，如果有微任务，就依次执行所有微任务，直到清空微任务队列。
    3. 然后取出宏任务队列中的第一个宏任务，执行宏任务。
    4. 然后再次检查微任务队列，重复第二步。
    5. 然后检查是否需要进行 UI 渲染，如果需要，就执行 UI 渲染。
    6. 然后再次取出宏任务队列中的下一个宏任务，重复第三步至第五步，直到宏任务队列清空。
